Mizan · ميزان

A calm, fast, multilingual system for the people running restaurants.

Mizan is the design language behind a tablet-first POS, kitchen display, and back-of-house suite built for the Moroccan market. Optimised for long shifts, bright daylight terraces, dim evening dining rooms, and the seamless French ↔ Arabic switch every team here lives in.

FR · AR · Darija Tablet-first Light + Dark WCAG AA Tokens + Tailwind
§ 01

Principles

Seven things we believe about how this product should feel under a Friday-night rush.

01

Calm under fluorescent light

The interface stays legible at a glance in the kitchen, on a sunlit terrace, and in a dim evening dining room. No decoration distracts from the task in front of the user.

02

The hand finds the button

Touch targets are at least 44px everywhere, 64px on the POS terminal. Primary actions live where the thumb expects them. Spacing is generous on purpose.

03

Status is shape, not just color

Order, table, and stock states always pair a color with a shape, icon, or label. A color-blind cashier and a tired manager see the same signal.

04

French and Arabic are equal

RTL is built on logical properties from day one. Numerals stay LTR, layouts flip cleanly, type metrics match across scripts. Darija lives in the same field FR does.

05

Instant or it’s broken

Tap responses fire in under 100ms. No animation in the POS exceeds 200ms. The system feels like wood, not like the web.

06

Warm, not folkloric

A whisper of terracotta and sand in the neutrals signals “made here” without leaning on motifs. No zellige, no lanterns, no calligraphy as ornament.

07

Density follows the surface

The POS breathes. The manager dashboard packs information. The recipe builder is a working form. Each surface has its own density tier, drawn from the same tokens.

v1 ships comfortable only.

Dense is defined as a future feature for established restaurants whose staff have memorised the menu and want more items per screen. We are not exposing a toggle in v1 — the system is locked in comfortable to enforce a single, predictable training experience. Section 4 below shows the same POS screen at both densities so PM can pressure-test the dense rendering before we commit to launching it.

To preview dense anywhere, add data-density="dense" on any container.

Dense graduation criteria. v3 lock

Dense mode ships in v1.1, post-pilot. Three conditions must all hold before it leaves the feature-flag:

  1. a.3–6 months of stable comfortable mode in production. We need a real baseline of tap-error rates, order-time, and complaint volume before we change tactile behaviour.
  2. b.Module 14 (Settings infrastructure) is built. Dense requires a per-tenant settings surface that doesn’t exist yet — we will not bolt a hidden toggle onto a config file in the meantime.
  3. c.At least one pilot restaurant has explicitly asked for denser views. If nobody is asking, we’ve solved a problem that doesn’t exist. Comfortable wins by default.
Two non-negotiables.
  • · Never default-on globally. Dense never becomes the system default — not in any version, not for any reason.
  • · Per-restaurant, not per-user. Settings live on the tenant. A new hire and a 5-year veteran must see the same density on the same terminal during the same shift.

Reasoning: new user + dense = error-rate disaster. A server on day one cannot safely operate a 48px-target POS during a Friday rush — the error rate spikes, orders mis-fire, frustration compounds. Comfortable is the training-wheels density and stays the default forever. Dense is a privilege earned by a kitchen that has clearly stabilised, and the per-tenant scope guarantees every shift on that terminal carries the same muscle memory.

§ 05

RTL · how the system flips

Arabic isn’t a translation layer — it’s a first-class direction. Switching dir="rtl" on <html> is enough; the rest is enforced by tokens, not per-screen handwork.

Flips automatically

  • · Layout properties: margin-inline-start/end, padding-inline-*, inset-inline-*, border-inline-*
  • · Flex / grid order with dir="rtl" on the document or container
  • · Drawer / sheet corner radii (mirrored via [dir="rtl"] overrides)
  • · Text alignment (text-align: start/end)
  • · Tab and segmented-control order
  • · Avatar status indicator position (corner)

Icons that need a flip

  • · Directional arrows (arrow-left/right, chevron-*)
  • · Send / forward / reply
  • · Back navigation, breadcrumb chevrons
  • · Undo / redo
LTR RTL

Stays unchanged

  • · Numerals · always LTR with tabular-nums
  • · Prices, SKUs, phone numbers, timestamps
  • · Brand mark and logo
  • · Non-directional icons (check, x, plus, search, settings)
  • · Media playback controls (play, pause, stop)

Implementation rule

Engineers should never write margin-left or padding-right. Always use logical equivalents: margin-inline-start, padding-inline-end. If you do reach for a physical property, it’s a smell — wrap it in a [dir="ltr"] override and write a comment explaining why. Numerals get font-variant-numeric: tabular-nums on every container, and any mixed-script line (an Arabic sentence with a price) wraps the LTR fragment in <span dir="ltr"> to keep ordering deterministic.

§ 06

Accessibility

Restaurant staff vary widely in age, eyesight, and color perception. The system bakes in WCAG AA from the start, not as a polish step.

Contrast

All text/background pairs pass WCAG AA (4.5:1) in both themes. Key tested pairs:

fg / bg (light)15.6 : 1AAA
fg-2 / bg (light)7.1 : 1AAA
fg-3 / bg (light)4.6 : 1AA
primary / surface5.2 : 1AA
fg / bg (dark)14.8 : 1AAA
primary (dark) / surface4.7 : 1AA

Focus visible

A 3px terracotta ring on every focusable element, applied via :focus-visible so it only shows for keyboard users — taps don’t flash the ring on tablets.

Type scale to 200%

All type sizes survive a browser-level zoom to 200% without breaking layout: containers reflow, touch targets remain ≥ 44px, and no labels truncate. POS-specific surfaces ship a built-in “Large” mode for staff with vision needs.

Color-blind safety

Color is never the only signal. Every status pairs a hue with an icon, a shape, or a label:

Servi En cuisine En retard En attente

Verified across deuteranopia, protanopia, and tritanopia simulations. Floor-plan tables additionally use border-style (solid / dashed / striped) so state survives any color-vision deficit.

For engineering
  • Token contract lives in mizan-tokens.css at the project root. That file is the single source consumed by apps/admin-web (later) and apps/pos-terminal (in progress).
  • Import: @import "mizan-tokens.css"; at the top of your global stylesheet. Read tokens via var(--*); never hard-code values.
  • Do not write color: var(--accent-celebration) directly. Use the saffron utility classes. Adding new ones requires PR review.
  • Do not hard-code tap-targets. Use var(--density-tap-target), var(--density-row-padding), etc. The density swap depends on it.
